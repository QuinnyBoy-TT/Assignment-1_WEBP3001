Logical View

Functional Requirements in this application are ability to provide administrative authority to employees and other information Authentication for customers who purchase products Authentication for select employees who handle the company aspect. Perform SCRUD (Search Create Retrieve Update and Delete {archive}) on Customer Orders, Inventory Management, Staff Schedule Management, Staff Information management, keeping track of records monthly and generating excel spreadsheets, Creating an E-Commerce aspect of the web application for the advertisement and business side of the project. In this application Subsystems can be described as a three-tier layered system. These systems are:

  1. User management subsystem
  2. Staff Management subsystem
  3. Advertisement management subsystem

These layers play a very important role amongst one another, both user and advertisement management subsystem can be managed by the staff management subsystem but not the other way around.

Process View

Non-Functional Requirements in this application are Scalability which would be considered adding in more e-commerce as well as a mobile and desktop application that can coexist with the Django and Python framework. Supportability which are the system needs to be cost-effective to be maintained for a minimal cost, majority of the structure and modules were created within a framework as to reduce the cost and time it takes to develop a product. Reliability in terms of users have to trust the system and even after using it for a long time. They have to trust the fact that the system has a good fault tolerance and can rebound from any potential fault points or failure. Along with Portability as in the choices and flexibility in technology is now a choice because of the decision to make this application a web-based application as well as manage it on the cloud. It also means users and staff are also able to access the application from a mobile or any other preferred platform without compatibility issues. In terms of concurrency and synchronization in this application , this can be very apparent from a technical aspect, when logging in to an authorized account a user is granted a session and this cannot be broken until the user logs out of the system , this session provides access to their instance of record keeping and generating , if this user does any form of data processing there is usually a delay between processing the query and not processing the query as to reduce overlapping data queries.

Development View

As previously touched on in the logical view, the subsystem consists of three layers, this would be described in detail later on in the technical document. By using Django, the modules used in creating these layers can be described by talking about its components, the top layer consist of a URL Patterns. This works by running through each URL pattern, in order, and stops at the first one that matches the requested URL, matching against path info. Once one of the URL patterns matches, Django imports and calls the given view, which is a Python function (or a class-based view).There’s is then a View layer, a view function, or view for short, is a Python function that takes a Web request and returns a Web response. This response can be the HTML contents of a Web page, or a redirect, or a 404 error, or an XML document, or an image . . . or anything, really. The view itself contains whatever arbitrary logic is necessary to return that response. This code can live anywhere you want, as long as it’s on your Pythonpath.The layers then extend to a Model which is the single, definitive source of information about your data. It contains the essential fields and behaviors of the data you’re storing. Generally, each model maps to a single database table. The final part of this is Templates and this document describes Django’s built-in template tags and filters.

Physical View

Processes are mainly allocated to mainly the server and framework, minimal processing has to be done on the physical device , the way how this application was designed with compatibility , performance ,scalability and maintainability in mind. This being said Django ensures this this web application is not only desktop friendly but also mobile friendly in its operations. Django aids in splitting up the different components into majority of its bulkier processes be handled server side. As explained further into the technical document this framework follows a modified version of a MVC(Model View Controller) it is called MVT(Model ,View ,Template).Both the View and the Model are handled by the webserver making it easier on hardware.
